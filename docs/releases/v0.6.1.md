# Omnigraph v0.6.1

`omnigraph optimize` no longer crashes on a graph that has a `Blob` property. Lance `compact_files` mis-decodes blob-v2 columns under its forced `BlobHandling::AllBinary` read (`Invalid user input: there were more fields in the schema than provided column indices / infos`), which previously aborted the entire optimize sweep. Reads and writes were never affected — only compaction.

## Behavior changes

- **Blob-bearing tables are skipped during `optimize`** instead of crashing it. Such a table is reported with `TableOptimizeStats.skipped = Some(SkipReason::BlobColumnsUnsupportedByLance)` — surfaced in the CLI (`optimize` human output and a new `skipped` field in `--json`) and a `tracing::warn` — and the rest of the sweep proceeds, so one blob table no longer aborts compaction for the healthy tables. Gated behind the `LANCE_SUPPORTS_BLOB_COMPACTION` constant, which flips when the upstream Lance fix lands.
- **Consequence:** fragment count and deleted-row space on blob tables are not reclaimed until that Lance fix; query results, reads, and writes are unaffected. No on-disk migration — existing graphs open unchanged.

## Tests added

- `tests/maintenance.rs::optimize_skips_blob_table_and_reports_skip` — a blob table is skipped and reported while a sibling non-blob table still compacts.
- `tests/lance_surface_guards.rs::compact_files_still_fails_on_blob_columns` — self-retiring guard that turns red on the Lance bump that fixes the bug, forcing `LANCE_SUPPORTS_BLOB_COMPACTION` to flip.

## Files changed

- `crates/omnigraph/src/db/omnigraph/optimize.rs` — `LANCE_SUPPORTS_BLOB_COMPACTION` gate, `SkipReason`, `TableOptimizeStats.skipped`, and the blob-table skip in `optimize_all_tables`.
- `crates/omnigraph-cli/src/main.rs` — render the skip in `optimize` output.
